103  class Condition : public Mutex {
104  public:
105  /*
106  The wait() and wait( int millis )g functions are used to block
107  on a condition variable. They are called with mutex locked by the calling thread
108  or undefined behaviour will result.
109  These functions atomically release mutex and cause the calling thread to block on
110  the condition variable cond; atomically here means "atomically with respect to access
111  by another thread to the mutex and then the condition variable". That is, if another
112  thread is able to acquire the mutex after the about-to-block thread has released it,
113  then a subsequent call to pthread_cond_signal() or pthread_cond_broadcast() in that
114  thread behaves as if it were issued after the about-to-block thread has blocked.
115 
116  Upon successful return, the mutex has been locked and is owned by the calling thread.
117 
118  When using condition variables there is always a boolean predicate involving shared variables
119  associated with each condition wait that is true if the thread should proceed. Spurious wakeups
120  from the wait() or wait( int millis )g functions may occur. Since the return
121  from wait() or wait( int millis )g does not imply anything about the value
122  of this predicate, the predicate should be re-evaluated upon such return.
123 
124  The effect of using more than one mutex for concurrent wait() or
125  wait( int millis )g operations on the same condition variable is undefined; that is, a
126  condition variable becomes bound to a unique mutex when a thread waits on the condition variable,
127  and this (dynamic) binding ends when the wait returns.
128  */

131  int wait ( int millis = 0);
132  int waitAndLock ( int millis = 0 );
